Power-Up Guidelines

This is a resource to help guide you in building your Power-Up.

Guiding Principles

  • Quality over quantity.
  • Don’t embed your site.
  • Don't rely on or prompt for a browser extension for full functionality. Power-Ups should stand on their own.
  • Shared Perspective - when and where possible give a consistent view to all board members.
  • Ensure your Power-Up works across the same browsers Trello does. Supported Platforms.

Capabilities

Attachment Sections

Dos:
  • Use this to show additional information about an attachment that your service knows about.
  • Make it obvious what attachments have been pulled out into your sections.
  • Make it clear how the user could navigate to the original content at the URL(s) that were claimed for the section.
  • Names for attachment sections should be unique and easy to understand.
Don’ts:
  • Use attachment-sections when the information you are adding can be handled by attachment-thumbnails.
  • Re-render the entire section if an item is added to or removed from it, this leads to flashing.
  • Use more vertical space than is required for the content.

Attachment Thumbnails

Dos:
  • Use this to show additional structured information about an attachment your service knows about
Don’ts:
  • Respond to attachment-thumbnail requests for attachments your integration doesn’t logically provide additional useful information for.

Board Buttons

Dos:
  • Use this when you have common actions that are applicable to the entire board.
Don’ts:
  • Provide more board buttons than is necessary, this clutters the header space.
  • Provide a board button that has the same functionality as a card button.
  • Provide a colored icon.

Card Buttons

Dos:
  • Use when you have actions that are applicable at a card level, such as searching for and attaching content to a card.
Don’ts:
  • Provide more buttons than is necessary. Most Power-Ups need only one button. If necessary use a single button that opens a popup with a list of further options.
  • Provide a colored icon.

Card Badges

Dos:
  • Use to provide high level context on cards, individually relevant to the cards.
  • Create badges that provide at-a-glance information that is useful to the user.
Don’ts:
  • Provide badge content that is so long it clips.
  • Refresh on an interval shorter than is necessary.
  • Use badge colors when not logical.
  • Provide more badges than necessary. Most Power-Ups only require adding a single badge to cards.

Card Detail Badges

Dos:
  • Use to provide high level context on cards, individually relevant to the cards.
  • Provide a popup or a url for click the badge where logical (see: custom fields).
  • Provide a sensible and short title for each badge.
Don’ts:
  • Abuse colors where not logical.
  • Provide more badges than necessary.
  • Refresh on an interval shorter than is necessary.

Card From URL

Dos:
  • Use to create new cards from URLs that your service knows about, especially if the information requires some form of authentication.
Don’ts:
  • Create cards for URLs not obviously associated with your Power-Up.
  • Clutter the card by adding more content to the description or title than makes sense and is necessary.

Format URL

Dos:
  • Try to account for the different variations a url may have to represent a single entity. /item/123 and ?item=123 may represent the same item in some cases.
  • Provide an appropriate icon with the URL where possible, especially for branding purposes (make sure you own the copyright where applicable)
Don’ts:
  • Return the URL largely as is, but just with an icon. Your Power-Up should add value.
  • Return content for URLs that aren’t logically associated with your Power-Up functionality (the Google Drive Power-Up doesn’t try to format Dropbox URLs).
  • Return a colored icon.

Show Settings

Dos:
  • Less is more, keep the number of settings / preferences to a minimum.
  • Provide helpful details and information about any settings.
  • Provide the ability to edit settings once they have been configured.
Don’ts:
  • Use this for content or functionality that are core to the Power-Up. Your Power-Up should function well by default without the user ever needing to go here.

Controls

In General:

Dos:
  • Keep it clean and simple.
  • Performance is important, try to make your experiences feel “native.”
  • Use Trello’s provided CSS: https://trello.com/power-ups/power-up.css
Don’ts:
  • The Power-Up's functionality should include more than just linking a user to your site.
  • Override Trello’s CSS except where necessary, for example to preserve certain band identity.
  • Don't inject a chat/messanger widget or other tools that disrupt the Trello experience.

Popup

Dos:
  • Prefer this control over overlay, whenever possible because it maintains context better.
  • Minimize the number of height resizes you do.
  • When using a popup for search, make use of the built-in search functionality instead of using your own custom search iframe.
Don’ts:
  • Have horizontally scrolling content in the popup.
  • Have vertically scrolling content in the popup (except where long lists make sense).
  • Have a background color for your iframe.

Board Bar

Dos:
  • Resize to the perfect height for your content.
  • Keep the height as minimal as possible for the content.
Don’ts:
  • Make the content in the board bar distracting from the board. A board bar should not be so large that it covers the entire Trello web view.
  • Open a board bar arbitrarily. A board bar should only be opened in response to a user's action.

Overlay

Dos:
  • Make it easy to close, and obvious how to close.
  • Keep it as small as possible - both vertically and horizontally.
Don’ts:
  • Cover the entire Trello board.

General Topics

Data Storage

Dos:
  • Make use of 
    t.set
    and 
    t.get
    for storing simpler data such as settings/preferences.
  • When storing more complex data, store a reference to it externally with 
    t.set
    .
  • Use the ‘private’ scope when storing confidential or sensitive information, such as tokens.
Don’ts:
  • Store data in other Trello fields like card descriptions or comments.
